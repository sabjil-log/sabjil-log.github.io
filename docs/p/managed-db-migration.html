❯ 관리형 DB로 옮기면 끝인 줄 알았다 — 이관할 때 놓치는 것들

한 줄 요약: 관리형 DB(RDS, Cloud DB)가 대신해주는 건 서버 관리(설치·패치·하드웨어·기본 백업)입니다. DB 관리(파라미터, 백업 검증, 장애조치 설계, 권한, 성능)는 여전히 우리 일이에요. 이 경계를 모르면 "관리형인데 왜 터지죠"가 됩니다.

착각의 구조

"관리형"이라는 말이 주는 인상: 이제 DB는 신경 안 써도 된다. 현실: 건물 관리인이 생긴 것이지 집사가 생긴 게 아닙니다. 청소·수도·전기(인프라)는 해주지만, 집 안 가구 배치와 금고 비밀번호(설정과 데이터)는 여전히 집주인 몫이에요. 이관 때 놓치는 것들이 정확히 이 경계선 위에 있습니다.

놓치는 것 ① — 파라미터 그룹: my.cnf는 어디 갔나

기존 서버에선 my.cnf / postgresql.conf를 직접 만졌지만, 관리형에선 그 파일에 접근할 수 없습니다. 대신 파라미터 그룹이라는 콘솔 개념으로 바뀌었을 뿐 — 기본값으로 두면 된다는 뜻이 아닙니다.

단골 사고: 기존 서버에서 max_connections를 500으로 올려 쓰고 있었는데, 이관 후 기본 파라미터 그룹(인스턴스 크기 기준 자동 산정)이 그보다 작아서 피크 때 연결 거부. 문자셋(utf8mb4), 타임존, slow query 임계값도 같은 계열입니다.

-- 이관 전, 기존 DB에서 지금 설정을 떠두세요 (MySQL 예)
SHOW VARIABLES WHERE Variable_name IN
 ('max_connections','character_set_server','time_zone','long_query_time');

이 목록과 새 파라미터 그룹을 대조하는 것이 이관 체크리스트 1번입니다. 그리고 파라미터 중엔 적용에 재시작이 필요한 것(static)이 있으니, 변경 타이밍도 계획에 넣으세요.

놓치는 것 ② — 백업: '켜져 있음'과 '복구됨'은 다르다

자동 백업이 기본으로 켜져 있으니 안심합니다. 두 가지를 확인 안 한 채로요.

  • 보존 기간: 기본값이 짧게 잡힌 경우가 많습니다(며칠 수준). "지난달 데이터로 돌려주세요"가 불가능한 설정일 수 있어요. 규정·업무 요건에 맞게 명시적으로 설정해야 합니다.
  • 복구 리허설: 백업의 존재와 복구 가능은 다른 문제입니다. 관리형의 복구는 보통 기존 인스턴스를 덮는 게 아니라 새 인스턴스를 만드는 방식 — 복구 후 앱의 접속 엔드포인트를 바꿔줘야 한다는 뜻이죠. 이걸 장애 당일에 처음 알게 되면 복구 시간이 몇 배로 늘어납니다. 분기에 한 번, 스냅샷에서 새 인스턴스를 실제로 세워보는 리허설이 정답입니다.

놓치는 것 ③ — 고가용성: 다중화는 옵션이지 기본이 아니다

"클라우드니까 알아서 이중화되겠지" — 아닙니다. Multi-AZ / 고가용성 구성은 선택하고 돈을 내는 옵션입니다. 단일 구성으로 이관해놓고 가용성은 온프레미스 이중화 시절 기준으로 기대하면 사고죠.

그리고 다중화를 켰더라도 두 가지가 남습니다 — 장애조치(failover) 중 수십 초의 단절을 앱이 견디는가(커넥션 풀의 재연결 로직, 재시도), 그리고 앱이 개별 IP가 아닌 엔드포인트(DNS)로 접속하고 있는가. failover는 엔드포인트가 가리키는 대상을 바꾸는 방식이라, IP를 하드코딩한 앱은 다중화가 있어도 못 넘어갑니다.

놓치는 것 ④ — 접근 통제가 네트워크로 바뀐다

기존엔 DB 서버에 SSH로 들어가 로그 보고 파일 만졌지만, 관리형은 OS 접근 자체가 없습니다. 로그는 콘솔/API로 보고, 접근 통제는 DB 계정 + 네트워크(ACG/보안그룹, private 서브넷 배치) 두 겹으로 재설계해야 해요. 이관하면서 "일단 되게" 공인 접근을 열어놓는 게 최악의 패턴입니다 — private 서브넷에 두고 앱 대역만 여는 그 기본 구도(서브넷 글의 그림)로 가세요.

이관 당일보다 중요한 것 — 전환 계획

데이터 옮기기(덤프/복원 또는 복제 동기화)는 도구가 잘해줍니다. 실제 사고는 전환 순간에 나요. 최소 체크 세 가지 —

  1. 동기화 방식 결정: 중단 가능하면 덤프/복원, 무중단이면 복제(replication) 걸어 따라잡게 한 뒤 짧은 전환
  2. 전환 = DNS/설정 변경 한 곳으로: 앱들이 제각각 접속 정보를 들고 있으면 전환이 사냥이 됩니다. 미리 엔드포인트를 한 곳(설정/Secret Manager)으로 모아두세요
  3. 롤백 조건과 절차를 문서로: "언제 포기하고 되돌릴지"를 전환 전에 합의 — 새벽 3시에 정하는 게 아니라

한 줄 정리

관리형 DB는 서버를 없애주지 DBA를 없애주지 않습니다. 파라미터 대조, 백업 보존·복구 리허설, 다중화 옵션과 failover 대응, 네트워크 접근 재설계 — 이 네 가지가 이관 체크리스트의 진짜 본체예요.
