❯ 코드에 DB 비밀번호를 박으면 생기는 일 — Secret Manager를 쓰는 이유

한 줄 요약: 비밀번호를 코드에 넣는 순간 그 비밀은 git 히스토리·백업·개발자 노트북으로 무한 복제됩니다. Secret Manager는 비밀을 한 곳에만 두고, 앱이 실행 시점에 꺼내 쓰게 만들어 이 복제를 끊는 서비스예요.

흔한 시작

바쁘니까 일단 이렇게 짭니다.

# 이러지 마세요
DB_PASSWORD = "P@ssw0rd2026!"
conn = connect(host="db.internal", password=DB_PASSWORD)

돌아는 갑니다. 문제는 이 한 줄이 어디까지 퍼지느냐예요.

비유: 집 열쇠를 복사해서 뿌리기

코드에 박힌 비밀번호는 열쇠를 복사해서 나눠준 것과 같습니다.

  • git push 하는 순간 → 저장소를 클론한 모든 노트북에 열쇠 사본
  • 한 번이라도 커밋됐다면 → 나중에 지워도 git 히스토리에 영원히 남음
  • 저장소가 실수로 public이 되거나, 노트북 하나가 털리면 → 게임 끝

그리고 최악은 교체가 불가능해진다는 것. 비밀번호를 바꾸려면 코드를 고쳐 전 서비스를 재배포해야 하니, 다들 미루고, 같은 열쇠가 몇 년을 삽니다.

Secret Manager가 하는 일

발상을 뒤집습니다. 열쇠를 나눠주지 말고, 금고를 하나 두자.

  1. 비밀(DB 암호, API 키)은 금고(Secret Manager)에만 저장
  2. 앱은 실행될 때 "금고야, prod/db/password 줘" 라고 요청해서 받아 씀
  3. 금고는 누가(IAM), 언제, 뭘 꺼냈는지 전부 기록(감사 로그)
  4. 비밀번호 교체는 금고 값만 갱신 — 코드 수정·재배포 불필요, 주기적 자동 교체(rotation)도 가능

코드는 이렇게 바뀝니다.

# 개념 코드 — 비밀의 '이름'만 알고, '값'은 실행 시점에 받아온다
secret = secret_manager.get_secret("prod/db/password")
conn = connect(host="db.internal", password=secret)

코드 어디에도 실제 값이 없으니, 저장소가 통째로 유출돼도 비밀은 안전합니다.

"환경변수면 되잖아요?"

절반만 맞습니다. 환경변수는 코드와 비밀을 분리해주지만, 그 값은 결국 어딘가의 .env 파일이나 배포 스크립트에 평문으로 살아야 하죠. 열쇠 복사 문제가 장소만 옮긴 겁니다. .env를 실수로 커밋하는 사고는 너무 흔하고요. 환경변수는 "비밀의 이름이나 금고 주소"를 넘기는 용도로 쓰고, 값 자체는 금고에서 오게 하는 게 한 단계 위입니다.

NCP / AWS 둘 다 있다

  • AWS Secrets Manager — 저장·IAM 접근제어·감사·자동 rotation
  • NCP Secret Manager — 같은 개념. KMS 기반 암호화 저장, Sub Account 권한 제어와 조합

개념 지도는 같으니, 한쪽에서 익힌 습관이 그대로 이식됩니다.

이미 커밋해버렸다면

  1. 그 비밀은 유출됐다고 간주하고 즉시 교체가 1순위입니다. 히스토리 청소보다 먼저요.
  2. 저장소에 시크릿 스캐너(예: gitleaks)를 걸어 재발을 막습니다.
# 저장소 전체 히스토리에서 비밀 흔적 스캔
gitleaks detect --source . -v

한 줄 정리

비밀은 복사되는 순간 통제를 잃습니다. 코드에는 비밀의 이름만, 값은 금고(Secret Manager)에서 실행 시점에 — 그리고 이미 새어 나간 비밀은 지우는 게 아니라 바꾸는 겁니다.
