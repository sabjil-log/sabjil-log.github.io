❯ '포트는 열렸다는데 접속이 안 돼요' — 계층별로 찔러보는 진단법

한 줄 요약: "접속이 안 된다"는 하나의 문제가 아니라, DNS → 경로 → 포트 → 리스닝 → 앱 중 어느 한 층이 막힌 겁니다. 아래에서 위로 한 칸씩 짚으면 범인이 튀어나와요.

이런 적 있으신가요

curl은 타임아웃, 브라우저는 무한 로딩. 방화벽 담당자는 "포트 열려 있는데요?" 개발자는 "서버는 떠 있어요." 서로 핑퐁만 하다 30분 날립니다. 문제는 어디서 끊겼는지를 안 봤기 때문이에요.

비유: 택배가 안 왔을 때

  • 주소가 틀렸나? → DNS
  • 우리 동네까지 오긴 했나? → 네트워크 경로
  • 문이 잠겼나? → 방화벽/포트
  • 집에 사람이 있나? → 프로세스가 그 포트를 듣고 있나
  • 받고도 답을 안 하나? → 앱 레벨 오류

이 순서대로 한 층씩 확인하면, 끊긴 지점이 곧 범인 위치입니다.

1층 — 이름이 IP로 풀리나 (DNS)

dig +short example.com          # IP가 안 나오면 DNS 문제
# dig 없으면: nslookup example.com

2층 — 상대까지 닿나 (경로)

ping -c 3 example.com           # 단, ICMP를 막아둔 서버도 많음 → 실패해도 단정 금지
traceroute example.com          # 어느 홉에서 멈추는지

3층 — 그 포트가 열려 응답하나

nc -vz example.com 443          # succeeded → 포트 도달 OK
# nc 없을 때, 순수 bash 로도 가능:
timeout 3 bash -c '</dev/tcp/example.com/443' && echo OPEN || echo CLOSED

4층 — 내 서버가 정말 그 포트를 듣고 있나

서버에 들어가서(반대편이 내 서버라면):

ss -tlnp | grep ':443'          # LISTEN 상태와 어떤 프로세스인지 (netstat 후계자)

여기서 아무것도 안 나오면, 방화벽이 아니라 앱이 안 떠 있거나 다른 포트/주소에 바인딩된 겁니다. 127.0.0.1:443 으로만 듣고 있으면 외부에서는 못 붙어요 — 0.0.0.0 바인딩인지 확인하세요.

5층 — 방화벽 / 보안그룹

sudo iptables -L -n | grep 443  # 로컬 방화벽

클라우드면 인스턴스 방화벽(ACG / Security Group)과 서브넷 방화벽(NACL)을 함께 봅니다. NACL은 스테이트리스라, 들어오는 규칙만 열고 나가는 리턴 트래픽을 안 열면 응답이 되돌아가지 못해 "연결은 되는데 먹통"이 됩니다. (이건 다음 글에서 제대로 파요.)

6층 — 앱 레벨

3~5층이 다 통과인데 안 되면 앱 문제입니다.

curl -v https://example.com/health   # 상태코드/헤더/TLS 핸드셰이크까지 다 보임

500이면 앱 내부 오류, 연결은 됐다는 뜻. 그리고 최종 확인은 눈으로 패킷 보기:

sudo tcpdump -ni any port 443 and host example.com
# 패킷이 오가면 네트워크는 무죄, 안 오면 그 아래층 문제

한 줄 정리

접속 문제는 "되냐/안 되냐"가 아니라 "몇 층에서 막혔냐"입니다. dig → ping → nc → ss → 방화벽 → curl 순으로 아래부터 한 칸씩. 끊기는 지점이 곧 답이에요.
