❯ 배포했는데 예전 화면이 나와요 — CDN 캐시가 안 갱신되는 3가지 이유

한 줄 요약: CDN은 원본의 복사본을 약속된 기한(TTL)까지 나눠주는 편의점 체인입니다. 배포를 해도 옛 화면이 나오는 건 CDN이 고장 나서가 아니라 약속대로 아직 유효한 복사본을 주고 있는 것 — 해법은 TTL 설계, 무효화, 캐시 키 세 가지에 있습니다.

상황

새 버전을 배포하고 자신 있게 새로고침. 그런데 예전 화면이 그대로 나옵니다. 서버 로그를 보니 요청이 오지도 않았어요. 범인은 중간에서 복사본을 나눠주고 있는 CDN입니다.

비유: 본사 공장과 동네 편의점

  • 원본 서버(오리진) = 본사 공장
  • CDN 엣지 = 전국 편의점. 공장에서 물건(파일)을 받아다 두고 손님에게 팝니다.
  • TTL = 유통기한. "이 물건은 1시간 동안 그대로 팔아도 됨"이라는 약속.

배포는 공장의 물건을 바꾼 것이지, 편의점 진열대를 바꾼 게 아닙니다. 편의점은 유통기한이 남은 옛 물건을 계속 팔죠 — 계약대로요.

먼저: 정말 CDN이 범인인지 확인

curl -sI https://example.com/app.js | grep -iE "cache-control|age|x-cache|etag"
# Cache-Control: max-age=86400      ← 유통기한 하루짜리 약속
# Age: 4520                          ← 이 복사본이 만들어진 지 4520초
# X-Cache: Hit from cloudfront       ← 편의점(엣지)에서 나온 물건이라는 증거

X-Cache: Hit(벤더마다 헤더명은 조금씩 다름)에 Age가 크면 확진 — 엣지의 오래된 복사본입니다. Miss인데도 옛날 화면이면 CDN이 아니라 브라우저 캐시나 배포 자체를 의심하세요.

해법 ① — 급한 불: 무효화(invalidation)

편의점 전 지점에 "그 물건 회수!" 공문을 보내는 겁니다. 콘솔이나 CLI에서 경로를 지정해 캐시를 강제로 버리게 합니다.

# CloudFront 예시 (NCP CDN+ 등도 콘솔/API에 동일 개념의 purge 존재)
aws cloudfront create-invalidation \
  --distribution-id EXXXXXX --paths "/app.js" "/index.html"

단점: 전파에 시간이 걸리고, 벤더에 따라 횟수 제한·비용이 있습니다. 응급처치지 운영 방식은 아니에요. 매 배포마다 /* 전체 무효화를 날리는 팀이 많은데, 그건 다음 방법을 모를 때의 임시방편입니다.

해법 ② — 근본: 파일명에 버전 박기 (cache busting)

발상 전환: 물건을 회수하지 말고 신제품은 새 이름으로 내놓는 겁니다.

app.js         →  기한 5분 (짧게)
app.3f9c2a.js  →  기한 1년 (사실상 영구)

빌드 도구(webpack/Vite 등)가 파일 내용의 해시를 파일명에 박아줍니다. 내용이 바뀌면 파일명이 바뀌고, HTML이 새 파일명을 가리키니 캐시 충돌 자체가 불가능해져요. 그래서 표준 설계는:

  • HTML (index.html): TTL 짧게 또는 no-cache — 늘 최신 목차를 받게
  • 해시 박힌 정적 자원(js/css/img): max-age=31536000, immutable — 1년짜리 유통기한

이렇게 하면 무효화가 거의 필요 없어집니다. 배포 = 새 파일명 업로드 + HTML 갱신, 끝.

해법 ③ — 헷갈림 방지: 캐시 키

CDN은 "같은 요청인지"를 캐시 키(기본: URL, 설정에 따라 쿼리스트링·헤더·쿠키 포함)로 판단합니다. 여기서 두 방향의 사고가 납니다.

  • 키에 쿼리스트링이 포함 안 됨 → app.js?v=2로 부숴보려 해도 app.js와 같은 물건 취급 — 캐시 버스팅이 안 먹힘
  • 키에 불필요한 것까지 포함 → ?utm_source=... 가 붙을 때마다 딴 물건 취급 — 캐시 적중률 폭락, 오리진만 두들겨 맞음

캐시 키 설정을 열어 무엇으로 물건을 구분하는지 확인하세요. "무효화도 했는데 안 바뀐다"는 미스터리의 단골 범인이 캐시 키 불일치입니다. 그리고 마지막 관문인 브라우저 캐시도 같은 Cache-Control 약속을 따른다는 것 — CDN을 비웠어도 브라우저가 제 복사본을 쓰고 있을 수 있으니, 검증할 땐 시크릿 창이나 강력 새로고침으로요.

한 줄 정리

CDN은 약속(TTL)대로 일하고 있을 뿐입니다. 급하면 무효화, 근본은 해시 파일명 + HTML만 짧은 TTL, 미스터리는 캐시 키 확인. 이 셋이면 "옛날 화면" 문제는 설계 단계에서 사라집니다.
