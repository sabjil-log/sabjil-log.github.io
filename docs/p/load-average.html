❯ load average 4.0은 위험한가 — top 숫자를 읽을 줄 안다는 것

한 줄 요약: load average는 퍼센트가 아니라 "일하는 중 + 줄 서서 기다리는" 작업의 평균 개수입니다. 그래서 절대값이 아니라 CPU 코어 수와 비교해야 합니다. 4코어 서버의 4.0은 만석, 16코어 서버의 4.0은 한산한 거예요.

이런 장면

모니터링 알람: "load average 4.2!" 신입이 놀라서 달려옵니다. 그런데 그 서버가 16코어면 아무 일도 아닙니다. 반대로 2코어 서버의 4.2는 진짜 문제고요. 같은 숫자, 다른 의미 — 기준을 모르면 읽을 수 없는 지표입니다.

비유: 계산대와 줄

CPU 코어는 마트 계산대, 프로세스는 손님입니다.

  • 계산대 4개에 손님 4명 → 전원 계산 중, 대기 0. load = 4.0, 딱 만석.
  • 계산대 4개에 손님 8명 → 4명 계산, 4명 대기. load = 8.0, 과부하.
  • 계산대 16개에 손님 4명 → 널널. load = 4.0인데 여유.

즉 load ÷ 코어 수가 진짜 지표입니다. 1.0 근처면 꽉 채워 쓰는 중, 1.0을 계속 넘으면 일이 밀리는 중이에요.

읽는 법

uptime
# 14:02:33 up 41 days, ... load average: 2.10, 3.45, 4.80

nproc          # 코어 수 확인

숫자 세 개는 최근 1분 / 5분 / 15분 평균입니다. 셋의 관계가 추세를 말해줘요.

  • 2.10, 3.45, 4.80 — 1분 < 15분: 부하가 가라앉는 중 (방금 피크가 지나감)
  • 4.80, 3.45, 2.10 — 1분 > 15분: 부하가 차오르는 중 (지금 뭔가 벌어지고 있음)

함정: load가 높은데 CPU는 논다?

리눅스의 load에는 CPU 대기뿐 아니라 디스크/NFS 등 I/O를 기다리는(uninterruptible, D state) 프로세스도 포함됩니다. 그래서 "load 20인데 CPU 사용률 10%"라는 기묘한 상황이 나와요 — 범인은 CPU가 아니라 느린 디스크나 죽은 NFS 마운트입니다.

구분하는 법:

top            # 헤더의 %Cpu(s) 에서 us(유저)/sy(시스템)/wa(I/O대기) 확인
# wa 가 높다 → I/O 병목. CPU가 아니라 디스크/스토리지를 봐야 함

ps -eo state,pid,cmd | awk '$1=="D"'   # I/O에 붙들린(D state) 프로세스 색출
iostat -x 2                            # 어떤 디스크가 바쁜지 (%util 높은 장치)
  • us 높음 → 애플리케이션이 진짜 일을 많이 하는 것 (스케일업/아웃 고려)
  • sy 높음 → 커널이 바쁨 (과도한 syscall, 네트워크 처리 등)
  • wa 높음 → 디스크가 병목 (느린 볼륨, 폭주하는 로그, NFS 행)

htop에서 하나 더

htop의 CPU 막대에서 색이 의미를 갖습니다 — 초록(유저), 빨강(커널), 회색·주황 계열(I/O 대기 등). 막대가 꽉 찼는데 초록이 아니라면, 애플리케이션 코드 밖에서 원인을 찾아야 한다는 신호입니다.

한 줄 정리

load average는 줄 길이입니다. nproc으로 계산대 수부터 확인하고 load ÷ 코어 ≈ 1.0을 기준선으로, 1분/15분 비교로 추세를, wa와 D state로 CPU 문제인지 I/O 문제인지를 가르세요. 숫자 하나 무서워할 게 아니라 나눗셈 한 번이면 됩니다.
