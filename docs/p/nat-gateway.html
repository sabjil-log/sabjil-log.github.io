❯ NAT Gateway는 정확히 무슨 일을 하나 — 그리고 왜 요금 폭탄의 단골인가

한 줄 요약: NAT Gateway는 private 서버의 패킷에서 발신자 주소를 자기 공인 IP로 바꿔치기하고, 답장이 오면 장부를 보고 원래 주인에게 배달하는 번역 창구입니다. 시간당 요금에 처리한 데이터량 과금이 붙어서, 대용량 트래픽이 지나가면 조용한 요금 폭탄이 돼요.

복습 한 줄, 그리고 진짜 질문

서브넷 글에서 봤듯 NAT은 private 서버의 "나가기 전용 문"입니다. 오늘의 질문은 그다음 — 문 안에서 무슨 일이 벌어지길래 나가는 건 되고 들어오는 건 안 될까요?

원리: 회사 대표번호의 교환원

private 서버(10.0.3.15)가 외부 API(203.0.113.9)를 호출하면, NAT은 패킷을 이렇게 고칩니다.

[나갈 때 — 발신자 바꿔치기 (SNAT)]
원본:  from 10.0.3.15:44210  →  to 203.0.113.9:443
변환:  from NAT공인IP:5501   →  to 203.0.113.9:443
                └ 그리고 장부에 기록: "5501번 = 10.0.3.15:44210의 통화"

[답장 올 때 — 장부 조회 후 배달]
도착:  from 203.0.113.9:443  →  to NAT공인IP:5501
변환:  from 203.0.113.9:443  →  to 10.0.3.15:44210   (장부 보고 복원)

회사 교환원과 같습니다. 직원이 밖에 전화하면 상대에겐 대표번호만 보이고, 교환원은 "5501번 회선 = 김대리" 장부를 들고 있다가 답신을 연결하죠. 여기서 "들어오는 건 안 되는" 이유도 자명해집니다 — 장부는 나간 통화에 대해서만 생기니까. 외부에서 먼저 걸려온 전화는 장부에 없어서 연결할 곳이 없는 겁니다. NAT의 단방향성은 기능이 아니라 이 구조의 자연스러운 결과예요.

부수 효과 둘: 외부 서비스 입장에선 우리 서버 수십 대가 전부 NAT의 IP 하나(또는 몇 개)로 보입니다. 외부 API에 "우리 IP를 허용목록에 넣어주세요" 할 때 대야 하는 게 서버 IP가 아니라 NAT의 공인 IP인 이유. 그리고 장부(포트 매핑)는 유한해서, 한 목적지로 초대량 동시 연결을 만들면 포트 고갈이 여기서도 일어납니다 — TIME_WAIT 글의 그 문제가 NAT 층에서 재현되는 거죠. 해법도 같습니다: 커넥션 재사용.

비용 구조 — 왜 폭탄이 되나

NAT Gateway 요금은 대개 두 축입니다.

① 시간당 존재 요금   (떠 있기만 해도 매시간)
② 데이터 처리 요금   (지나간 GB당 — 방향 불문)

②가 함정입니다. NAT을 지나는 모든 바이트에 처리 요금이 붙는데, 이게 전송 요금과 별도로 얹혀요. 전형적인 폭탄 시나리오 —

  • private 서버들이 S3/Object Storage에서 대용량을 내려받는 파이프라인 (같은 클라우드 안인데도 NAT을 경유하면 처리 요금 발생)
  • 매일 수백 GB 로그를 외부 수집기로 전송
  • 컨테이너 노드들이 외부 레지스트리에서 이미지를 수시로 pull

전부 "인터넷 나가는 문 = NAT 하나"로 설계한 결과, 내부 창고 왕복까지 교환원 손을 거치며 건당 수수료를 무는 그림입니다.

우회로 — 같은 클라우드 안이면 전용 통로로

핵심 처방은 클라우드 내부 서비스로 가는 트래픽을 NAT에서 빼는 것입니다.

  • 게이트웨이형 엔드포인트: S3 등 주요 스토리지용. 라우팅 테이블에 "S3로 가는 건 이 전용 통로로"를 추가하면 NAT을 안 거칩니다. 추가 요금도 없는 경우가 많아 사실상 공짜 점심 — S3 트래픽이 큰 VPC라면 무조건 1순위.
  • 인터페이스형 엔드포인트(PrivateLink류): 그 외 API 서비스용. 시간당 요금이 있으니 트래픽이 클 때 NAT 처리비와 견줘 결정.
  • NCP도 같은 그림 — Object Storage 등으로 향하는 트래픽의 내부 경로 제공 여부를 확인하고 라우팅을 분리하는 접근은 동일합니다.

점검은 간단해요. NAT의 트래픽 모니터링을 열어 어디로 가는 트래픽이 큰지 보고, 상위가 같은 클라우드의 스토리지·서비스라면 엔드포인트 전환 후보입니다. 비용 누수 톱5 글의 "이 화살표는 유료인가?" 질문을 NAT에 정조준하는 셈이죠.

설계 메모 두 개

  • 가용성: NAT Gateway는 특정 존에 삽니다. 그 존이 죽으면 다른 존 서버들의 외부 통신도 같이 죽어요. 존별 NAT + 존별 라우팅으로 분리하는 게 프로덕션 정석 (비용과의 트레이드오프).
  • NAT 인스턴스라는 선택지: 직접 서버를 세워 NAT 역할을 시키면 쌉니다 — 대신 패치·장애·대역폭이 전부 내 일이 되죠. 트래픽 작은 개발 환경용 절약 카드 정도로 보면 됩니다.

한 줄 정리

NAT Gateway는 주소 바꿔치기 + 장부로 굴러가는 교환원이고, 장부 구조상 단방향입니다. 그리고 지나가는 GB마다 수수료를 받으니 — 내부 스토리지행 대용량 트래픽은 엔드포인트 전용 통로로 빼는 것, 이게 NAT 비용 최적화의 알파이자 오메가예요.
