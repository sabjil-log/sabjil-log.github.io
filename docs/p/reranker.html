❯ 1차 검색은 왜 부족한가 — 리랭커가 RAG에서 하는 일

한 줄 요약: 벡터 검색(1차)은 빠르지만 대충 고르고, 리랭커(2차)는 느리지만 꼼꼼히 다시 줄 세웁니다. 채용으로 치면 서류전형 → 면접 — 이 2단 구조가 RAG 검색 품질의 표준 레시피예요.

증상: 분명 문서에 있는데 엉뚱한 게 뽑힌다

RAG를 돌려보면 이런 일이 생깁니다. "환불 수수료가 얼마야?"라고 물었는데, 뽑혀온 청크는 "수수료 관련 문의가 많습니다" 같은 비슷하지만 답이 없는 조각들. 답이 든 조각은 5등, 8등쯤에 밀려 있고요. 임베딩 검색의 태생적 한계입니다.

왜 1차 검색은 거친가

임베딩 검색(bi-encoder)은 질문과 문서를 각자 따로 벡터로 만들어 놓고, 거리만 잽니다. 문서 벡터는 질문이 뭔지 모르는 상태에서 미리 만들어진 요약이에요.

비유하면 서류전형입니다. 지원서(문서 벡터)는 회사가 뭘 물어볼지 모른 채 미리 써둔 것이고, 심사관은 지원서 요약본만 훑어 1초에 수천 장을 거릅니다. 빠른 대신, "질문과 딱 맞물리는지"는 못 봐요. 그래서 어휘가 비슷한 오답이 상위에 섞입니다.

리랭커는 뭘 다르게 하나

리랭커(cross-encoder)는 질문과 문서를 한 쌍으로 같이 모델에 넣고, "이 문서가 이 질문에 답이 되는가"를 직접 채점합니다.

이건 면접입니다. 질문을 실제로 던져보고 반응을 보는 것. 정확하지만 한 쌍마다 모델을 한 번씩 돌려야 해서 느리죠. 그래서 전체 문서에는 못 쓰고, 서류전형 통과자에게만 씁니다.

2단 파이프라인

질문 → [1차: 벡터 검색] 수만 개 중 top-50 후보   (빠름, 거침)
     → [2차: 리랭커]   50개를 재채점해 top-5     (느림, 정확)
     → LLM에 top-5 만 근거로 전달

개념 코드로는 이렇습니다.

# 1차: 벡터 검색으로 넉넉히 후보 수집
candidates = vector_db.search(embed(question), k=50)

# 2차: 리랭커가 (질문, 문서) 쌍을 직접 채점
scores = reranker.score([(question, c.text) for c in candidates])
top5 = [c for c, s in sorted(zip(candidates, scores),
                             key=lambda x: -x[1])][:5]

answer = llm(prompt_with(question, top5))

핵심 감각 두 가지:

  • 1차는 넉넉하게 (k=30~100). 리랭커가 구제할 수 있게, 정답이 후보군에 들어오기만 하면 됩니다. 1차에서 k=5로 좁혀버리면 리랭커가 손쓸 재료가 없어요.
  • 2차는 야박하게 (top 3~5). LLM에 잡음을 덜 주는 게 답변 품질에 직결됩니다.

비용 감각

리랭커는 후보 개수만큼 모델 추론이 돕니다. 50개 재채점이면 추론 50번 — 그래도 문서 전체가 아니라 후보만이라 감당 가능한 수준이고, 보통 수십~수백 ms가 추가됩니다. 정확도가 중요한 사내 문서 QA라면 남는 장사고, 초저지연이 생명인 자동완성 같은 데선 뺄 수도 있는 트레이드오프예요.

언제 리랭커부터 의심하나

  • 답이 문서에 분명히 있는데 자꾸 못 찾는다 → 1차 상위권 잡음 → 리랭커 도입
  • 리랭커를 넣었는데도 못 찾는다 → 후보군(k)을 늘리거나, 애초에 청킹이 문제일 가능성 (완결된 생각 단위로 잘렸는지부터 재점검)

한 줄 정리

벡터 검색은 서류전형, 리랭커는 면접입니다. 넉넉히 거르고(1차 k=50), 꼼꼼히 줄 세워(2차 top-5) LLM에는 정예만 넘기세요. RAG 답변 품질이 한 계단 다르게 올라갑니다.
