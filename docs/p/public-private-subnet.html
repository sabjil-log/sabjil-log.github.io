❯ public 서브넷과 private 서브넷, 뭐가 다른 걸까 — 라우팅 테이블이 전부다

한 줄 요약: public/private은 서브넷에 붙은 딱지가 아니라 라우팅 테이블의 한 줄 차이입니다. 0.0.0.0/0 → 인터넷 게이트웨이 경로가 있으면 public, 없으면 private. 그게 전부예요.

왜 나누나

건물로 비유하면 VPC는 회사 사옥, 서브넷은 층입니다.

  • public 서브넷 = 1층 로비. 외부 손님이 드나드는 층 — 로드밸런서, 배스천(점프) 서버가 삽니다.
  • private 서브넷 = 카드 찍어야 들어가는 사무층. 외부에서 직접 못 들어옵니다 — WAS, DB가 삽니다.

DB를 로비에 두는 회사는 없죠. 공격 표면을 줄이기 위해 인터넷을 직접 만나야 하는 것만 public에 두고, 나머지는 전부 private에 넣는 게 기본 설계입니다.

그런데 "public"이라는 설정은 없다

콘솔에서 서브넷을 만들 때 public/private 체크박스가 있긴 하지만(NCP처럼), 본질은 그 서브넷이 어떤 라우팅 테이블을 쓰느냐입니다. 라우팅 테이블은 "이 목적지로 가려면 어느 문으로 나가라"는 안내판이에요.

# public 서브넷의 라우팅 테이블
10.0.0.0/16   → local            (VPC 안은 서로 직통)
0.0.0.0/0     → igw-xxxx         (그 외 전부 = 인터넷 게이트웨이로)

# private 서브넷의 라우팅 테이블
10.0.0.0/16   → local
(끝. 인터넷으로 가는 길 자체가 없음)

0.0.0.0/0(기본 경로)이 인터넷 게이트웨이(IGW) 를 가리키면 public, 그 줄이 없으면 private. 서브넷의 정체성은 이 한 줄이 결정합니다.

private 서버도 밖엔 나가야 한다 — NAT

private에 있는 WAS도 패키지 설치나 외부 API 호출은 해야죠. 그래서 NAT Gateway를 씁니다. 사무층 직원이 "대표번호"로 외부에 전화 거는 것과 같아요 — 나가는 건 되지만, 밖에서 그 번호로 직원 자리에 직접 걸 수는 없습니다.

# private 서브넷 + NAT 조합
10.0.0.0/16   → local
0.0.0.0/0     → nat-xxxx          (밖으로 나갈 땐 NAT을 통해서만)

IGW는 양방향 문, NAT은 나가기 전용 문이라는 게 핵심 차이입니다. (NAT Gateway 자체는 public 서브넷에 놓입니다 — 대표전화기가 로비에 있는 셈이죠.)

서버에서 직접 확인하기

# 이 서버가 인터넷에 나갈 수 있나
curl -m 5 -s https://ifconfig.me && echo || echo "밖으로 못 나감"
# 나갈 수 있다면, 찍히는 IP가 NAT의 공인 IP인지 내 공인 IP인지로
# private(NAT 경유)인지 public(직접)인지도 구분됩니다

ip route        # 서버 입장의 라우팅 확인 (기본 게이트웨이가 어디인지)

흔한 함정

  • private 서버에 공인 IP만 붙이면 되는 줄 안다 — 라우팅 테이블에 IGW 경로가 없으면 공인 IP가 있어도 통신이 안 됩니다. IP보다 경로가 먼저예요.
  • NAT 없이 private에 서버를 넣고 "yum이 안 돼요" — 당연합니다. 나가는 문이 없으니까요. NAT을 달거나, 내부 미러/엔드포인트를 쓰세요.
  • 다 public에 넣는 설계 — 편하지만 DB까지 인터넷에 노출하는 셈. 방화벽이 한 겹 실수하는 순간 사고가 됩니다.

한 줄 정리

서브넷의 public/private은 라우팅 테이블의 0.0.0.0/0 한 줄이 정합니다. 인터넷을 만나야 하는 최소한만 public(IGW), 나머지는 private(+나가기 전용 NAT) — 이게 VPC 설계의 기본 뼈대예요.
