❯ cron이 새벽에 도는데 시간이 밀린다? — UTC 함정과 표현식 읽는 법

한 줄 요약: cron 표현식은 다섯 칸짜리 알람 다이얼입니다. 그리고 "새벽에 맞췄는데 낮에 돈다" 싶으면, 대개 서버·CI가 UTC 기준이라 내 시간과 어긋난 겁니다.

이런 적 있으신가요

"매일 새벽 3시에 돌게 해놨는데 왜 낮에 실행됐지?" GitHub Actions나 클라우드 스케줄러에서 특히 자주 겪습니다. 표현식은 맞는데 결과가 엉뚱하죠. 범인은 보통 타임존입니다.

cron 5칸 읽기

cron은 다섯 칸으로 "언제"를 적습니다. 왼쪽부터 분·시·일·월·요일이에요.

┌───── 분   (0-59)
│ ┌─── 시   (0-23)
│ │ ┌─ 일   (1-31)
│ │ │ ┌ 월   (1-12)
│ │ │ │ ┌ 요일 (0-6, 0=일요일)
│ │ │ │ │
* * * * *

예시로 감을 잡아보세요.

0 18 * * *     매일 18:00 정각
*/15 * * * *   15분마다
0 9 * * 1      매주 월요일 09:00
30 2 1 * *     매월 1일 02:30

* 는 "매 순간", */n 은 "n마다", 숫자는 "그때 딱"입니다.

UTC 함정

핵심입니다. GitHub Actions를 비롯한 많은 서버·CI는 UTC로 cron을 해석합니다. 한국(KST = UTC+9)에서 새벽 3시에 돌리고 싶다면, 그대로 0 3 * * * 라고 쓰면 안 돼요. 3 − 9 = −6, 즉 전날 18:00 UTC로 적어야 KST 새벽 3시에 돕니다.

0 18 * * *     ← 이게 KST 03:00

지금 기준 시각과 변환은 이렇게 확인합니다.

date -u                    # 현재 UTC 시각
TZ=Asia/Seoul date         # 같은 순간을 KST로
crontab -l                 # 리눅스에 등록된 내 cron 목록

몇 가지 더 조심할 것

  • 정시 보장 안 됨: GitHub Actions 크론은 부하에 따라 수 분 지연될 수 있습니다. "정확히 0초"를 기대하지 마세요.
  • 방치된 저장소: 커밋이 오랫동안 없는 저장소는 예약 워크플로우가 자동 비활성화될 수 있습니다.
  • 서머타임: UTC엔 서머타임이 없지만, 지역 타임존으로 스케줄하는 시스템은 DST 때 한 시간 튈 수 있어요. 한국은 DST가 없어 이 문제는 덜합니다.

한 줄 정리

cron은 분·시·일·월·요일 다섯 칸. 그리고 실행 시각이 밀리면 표현식을 의심하기 전에 "이 시스템이 UTC로 도나?"부터 확인하세요. 대부분 거기서 끝납니다.
