❯ TIME_WAIT가 수만 개 쌓였는데 괜찮은 걸까 — 소켓 상태 읽는 법

한 줄 요약: TIME_WAIT는 고장이 아니라 TCP가 연결을 안전하게 닫는 정상 절차입니다(닫은 뒤 잠깐 자리를 비워두는 것). 수만 개여도 대부분 무해 — 진짜 문제는 한 목적지로 나가는 연결이 많아 로컬 포트가 고갈될 때 하나뿐이고, 근본 해법은 커널 튜닝이 아니라 커넥션 재사용입니다.

상황

서버가 좀 느려진 것 같아 소켓을 봤더니:

ss -s
# TCP:   31245 (estab 210, closed 30890, timewait 30712, ...)

TIME_WAIT 3만 개. 놀라서 검색하면 "커널 파라미터를 바꿔라"는 글이 쏟아집니다. 잠깐 — 이게 문제인지부터 판별해야 합니다.

TIME_WAIT의 정체: 식당의 '치우는 중' 테이블

TCP 연결을 먼저 끊는 쪽은, 끊고 나서도 그 자리를 잠시(보통 60초) 비워둡니다. 식당에서 손님이 떠난 테이블을 바로 새 손님에게 주지 않고 잠깐 치우는 것과 같아요. 이유는 두 가지 —

  1. 네트워크를 헤매다 뒤늦게 도착하는 옛 패킷이 같은 자리의 새 연결에 섞이는 사고 방지
  2. 상대에게 보낸 마지막 인사(ACK)가 유실됐을 때 재전송할 수 있게

즉 TIME_WAIT는 TCP의 설계된 예의입니다. 트래픽이 많고 짧은 연결이 잦은 서버라면 수만 개가 쌓이는 게 자연스러워요. 메모리도 소켓당 얼마 안 먹습니다.

그럼 언제 문제가 되나 — 단 하나의 시나리오

내 서버가 클라이언트로서, 같은 목적지에 짧은 연결을 초당 수백 개씩 만들 때.

나가는 연결은 로컬 포트를 하나씩 씁니다(임시 포트, 기본 약 2.8만 개). 연결을 만들고 바로 끊기를 반복하면, 끊은 포트들이 60초씩 TIME_WAIT로 잠겨 새 연결에 쓸 포트가 동납니다. 증상은 명확해요 — connect: cannot assign requested address.

전형적인 범인: 요청마다 새 커넥션을 여는 코드로 뒷단 API·DB·Redis를 두들기는 애플리케이션. 웹서버가 손님을 받는 방향(인바운드)의 TIME_WAIT는 이 문제와 무관합니다 — 그건 손님 쪽 포트를 쓰니까요.

판별 명령

# TIME_WAIT가 '어디로 향한' 연결인지 집계 — 한 목적지에 몰려 있나?
ss -tan state time-wait | awk 'NR>1{print $4}' | sed 's/:[0-9]*$//' | sort | uniq -c | sort -rn | head
#   28714 10.0.3.15     ← 특정 백엔드로 폭주 중이면 위험 신호

# 임시 포트 범위 확인 (분모 파악)
cat /proc/sys/net/ipv4/ip_local_port_range

특정 목적지 하나가 수만 개를 차지하고 있고, 그 수가 임시 포트 범위에 육박하면 — 그때가 진짜 문제입니다.

해법 — 순서가 중요하다

① 근본: 커넥션 재사용 (keep-alive / 풀링). 연결을 만들고 버리지 말고 재사용하세요. HTTP 클라이언트의 keep-alive 세션, DB 커넥션 풀, Redis 풀 — 이거 하나로 TIME_WAIT 생성 자체가 수백 분의 일로 줄어듭니다.

# 나쁨: 요청마다 새 연결 → TIME_WAIT 양산
requests.get(url)          # 매 호출이 connect/close

# 좋음: 세션 재사용 → 연결 몇 개로 수만 요청 처리
s = requests.Session()
s.get(url)                 # keep-alive로 같은 연결 재사용

② 보조: 커널의 재사용 허용.

sysctl -w net.ipv4.tcp_tw_reuse=1   # 나가는 연결에 한해 TIME_WAIT 포트 재사용 허용 (안전)

③ 절대 금지: tcp_tw_recycle. 옛날 글들이 권하는 이 옵션은 NAT 뒤 클라이언트를 무작위로 끊어먹는 부작용 때문에 커널에서 아예 삭제됐습니다. 이걸 권하는 글은 그 글 전체를 의심하세요.

한 줄 정리

TIME_WAIT 수만 개는 대개 바쁘다는 증거지 장애가 아닙니다. 문제는 오직 아웃바운드 포트 고갈 — ss로 목적지 집계부터 하고, 답은 커널 만지작이 아니라 커넥션 풀/keep-alive입니다. 테이블을 빨리 치우려 하지 말고, 손님(연결)을 오래 앉혀두세요.
