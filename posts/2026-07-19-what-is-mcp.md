> **한 줄 요약:** MCP(Model Context Protocol)는 **AI와 도구 사이의 표준 콘센트**입니다. 도구를 MCP 서버로 한 번 만들어두면, 그 규격을 지원하는 **어떤 AI 클라이언트에든** 꽂아 쓸 수 있어요 — 도구 통합의 M×N 문제를 M+N으로 바꾸는 게 핵심 가치입니다.

## 문제: 어댑터 지옥

에이전트 글에서 봤듯, LLM에 손발을 다는 건 "도구 명세 + 실행 루프"입니다. 그런데 현실의 고통은 그다음이에요 — 사내 DB 조회 도구를 만들었다 칩시다. 이걸 챗 서비스 A에도, IDE 어시스턴트 B에도, 자체 에이전트 C에도 붙이려면? **셋 다 연동 방식이 달라서 어댑터를 세 번** 짭니다. 도구가 10개, 클라이언트가 4개면 통합 작업이 40번 — 이게 M×N 문제입니다.

전자기기 세계가 이미 푼 문제죠. 기기마다 제각각이던 충전 단자가 **USB라는 규격 하나**로 수렴하니, 기기 제조사는 USB만 달면 되고 충전기 제조사도 USB만 만들면 됩니다. MCP가 노리는 게 정확히 이것 — **도구는 MCP 서버로 한 번만, 클라이언트는 MCP 지원 한 번만.** M×N이 M+N이 됩니다.

## 구조: 서버, 클라이언트, 그리고 세 가지 제공물

```
[AI 앱(호스트)] ─ MCP 클라이언트 ─── 표준 프로토콜 ───→ [MCP 서버: DB 도구]
                 MCP 클라이언트 ─── 표준 프로토콜 ───→ [MCP 서버: 사내 위키]
                 MCP 클라이언트 ─── 표준 프로토콜 ───→ [MCP 서버: 모니터링]
```

- **MCP 서버** = 도구를 규격에 맞춰 내놓는 쪽. "나는 이런 기능이 있어"를 표준 형식으로 광고합니다.
- **MCP 클라이언트** = AI 앱 쪽 플러그. 서버 목록을 받아 모델에게 도구 명세로 전달하고, 모델의 실행 요청을 서버로 중계.

서버가 내놓을 수 있는 건 크게 셋입니다.

- **Tools(도구):** 모델이 호출하는 실행 가능한 함수 — "서버 상태 조회", "티켓 생성". 에이전트 글의 그 tool use가 이 규격 위에서 돌아가는 것.
- **Resources(자원):** 모델이 읽을 수 있는 데이터 — 파일, 문서, 조회 결과. RAG의 근거 공급구를 표준화한 셈.
- **Prompts(프롬프트):** 서버가 제공하는 정형화된 작업 템플릿 — "이 도구는 이렇게 쓰면 좋아요"를 세트 메뉴로.

연결 방식은 두 갈래 — 같은 머신에서 프로세스로 띄워 표준입출력(stdio)으로 붙는 로컬형, HTTP로 원격에 두는 서버형. 개인 도구는 전자, 팀 공용 도구는 후자가 일반적입니다.

## 만들어보면 별거 아니다 (개념 스케치)

MCP 서버의 본질은 "함수에 설명표를 붙여 규격대로 내놓기"입니다.

```python
# 개념 코드 — SDK를 쓰면 실제로도 이 정도로 단순합니다
from mcp.server import Server

server = Server("infra-tools")

@server.tool()
def get_server_status(host: str) -> str:
    """지정한 호스트의 리스닝 포트와 부하 상태를 조회한다.
    host: 내부 호스트명 (예: web-01)"""
    return check_status(host)          # 실제 로직
```

docstring이 곧 **모델이 읽는 도구 설명**이 됩니다 — 에이전트 글에서 강조한 "도구 명세가 절반"이 여기서도 그대로예요. 언제 쓰는지, 인자 형식, 쓰면 안 되는 경우까지 설명에 담으세요.

## 만들 때의 보안 체크리스트

콘센트를 표준화하면 **꽂기 쉬워진다는 게 곧 위험**이기도 합니다. 에이전트 안전 원칙이 MCP에선 이렇게 구체화돼요.

- **서버의 권한 = 노출 범위.** MCP 서버가 admin 자격증명으로 돌면, 그 서버에 붙는 모든 AI가 admin입니다. 서버 자체를 최소권한 계정으로 — IAM 글의 원칙이 다시 등장.
- **파괴적 도구는 분리·게이트.** 조회 서버와 변경 서버를 나누고, 변경류는 클라이언트 쪽 승인(사람 확인)을 전제로 설계.
- **신뢰할 서버만 꽂기.** 반대 방향의 위험 — 출처 불명의 MCP 서버를 꽂는 건 출처 불명 브라우저 확장을 까는 것과 같습니다. 도구 설명 자체에 악성 지시를 심는 공격(프롬프트 주입의 변종)도 보고돼 있어요.
- **감사 로그.** 어떤 클라이언트가 어떤 도구를 언제 호출했는지 서버 쪽에서 기록 — 표준 콘센트일수록 계량기가 필요합니다.

## 한 줄 정리

MCP는 AI 도구 연결의 **USB 규격**입니다 — 도구는 한 번만 만들고 어디에나 꽂는다. 서버의 본질은 "설명 잘 붙인 함수 묶음"이고, 성패는 에이전트 때와 같습니다: **좋은 도구 설명, 좁은 권한, 위험한 버튼 앞의 사람.**
