> **한 줄 요약:** 디스크에는 **용량**말고도 **inode(파일 개수 한도)**라는 별도의 자원이 있습니다. 용량이 남아도 inode가 바닥나면 파일을 못 만들어요. `df -h` 말고 **`df -i`** 를 봐야 잡히는 문제입니다.

## 증상

로그에 `No space left on device`. 그런데 `df -h` 를 치면 사용률 30%, 수백 GB가 남아 있습니다. "공간이 남는데 왜 못 쓴다는 거야?" — 리눅스 트러블슈팅의 고전 미스터리죠.

## 비유: 주차장의 두 가지 한계

주차장이 꽉 차는 방법은 두 가지입니다.

- **바닥 면적이 참** — 용량(GB)이 가득 찬 것. `df -h` 가 보여주는 것.
- **주차권이 떨어짐** — 자리는 남았는데 발급할 주차권이 없는 것. 이게 **inode 고갈**입니다.

inode는 파일 하나마다 발급되는 주차권(메타데이터 슬롯)이에요. 파일시스템을 만들 때 개수가 정해지고, **파일 크기와 무관하게 파일 1개 = inode 1개**를 먹습니다. 그래서 0바이트짜리 파일이라도 수백만 개가 생기면, 용량은 멀쩡한데 주차권이 동나요.

## 확인하는 법 — 딱 한 글자 차이

```bash
df -h     # 용량 사용률 (평소 보던 것)
df -i     # inode 사용률 ← 이걸 봐야 함
```

`df -i` 의 `IUse%` 가 100%면 확진입니다.

## 범인 찾기 — 파일이 무더기로 생긴 곳

inode 고갈의 범인은 거의 항상 **작은 파일을 무한 생성하는 무언가**입니다. 세션 파일, 캐시 조각, 메일 큐, 로테이트 안 된 로그 조각이 단골이죠. 디렉토리별 파일 개수를 세서 내려가면 잡힙니다.

```bash
# 1단계: 최상위에서 어느 디렉토리가 많은지
for d in /*; do echo -n "$d: "; find "$d" -xdev -type f 2>/dev/null | wc -l; done

# 2단계: 의심 디렉토리 안으로 파고들기 (예: /var)
for d in /var/*; do echo -n "$d: "; find "$d" -xdev -type f 2>/dev/null | wc -l; done
```

수십만~수백만 개가 나오는 디렉토리가 범인입니다. `/var/spool`, `/var/lib/php/sessions`, `/tmp`, 캐시 디렉토리를 먼저 의심하세요.

## 해결

**1. 범인 파일들 정리** — 개수가 너무 많으면 `rm *` 은 인자 한도로 실패할 수 있어, `find` 로 지웁니다.

```bash
# 예: 7일 지난 세션 파일 삭제 (경로는 실제 범인으로)
find /var/lib/php/sessions -xdev -type f -mtime +7 -delete
```

> ⚠️ `-delete` 는 되돌릴 수 없습니다. 먼저 `-delete` 빼고 실행해 목록을 눈으로 확인한 뒤 붙이세요.

**2. 재발 방지** — 범인이 로그/캐시라면 로테이션이나 주기 청소(cron, systemd-tmpfiles)를 겁니다. 애플리케이션이라면 파일을 무한 생성하는 로직 자체를 고쳐야 하고요.

**3. 근본 대응** — 원래 작은 파일이 수천만 개 생기는 워크로드라면, 파일시스템을 만들 때 inode를 넉넉히 잡거나(`mkfs.ext4 -N`), inode를 동적으로 쓰는 XFS 같은 파일시스템을 고려합니다. (기존 ext4 파티션은 inode 수를 나중에 못 늘립니다 — 재포맷 필요.)

## 한 줄 정리

"용량 남는데 no space" 는 십중팔구 inode입니다. **`df -i` 로 확인 → `find ... | wc -l` 로 범인 디렉토리 색출 → 정리 + 재발 방지.** 주차장은 바닥만 넓다고 되는 게 아니라 주차권도 있어야 합니다.
