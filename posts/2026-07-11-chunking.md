> **한 줄 요약:** RAG에서 문서를 자르는 단위(청크)가 검색 품질을 좌우합니다. **크면 잡음, 작으면 문맥 상실.** 정답 크기는 없지만, "한 청크 = 하나의 완결된 생각"이라는 기준과 **오버랩**이라는 안전장치가 있습니다.

## 왜 잘라야 하나

RAG는 질문과 비슷한 문서 조각을 찾아 LLM에게 건네주는 구조입니다. 그런데 문서를 통째로 임베딩하면 안 될까요? 안 됩니다. 임베딩은 조각 하나를 **벡터 하나**로 요약하는데, 100페이지를 벡터 하나에 욱여넣으면 온갖 주제가 평균돼 **아무것도 아닌 방향**을 가리키게 돼요. 그래서 자릅니다.

## 비유: 책을 인덱스카드로 만들기

책 내용을 카드에 옮겨 적어 검색용 카드함을 만든다고 해봅시다.

- **카드가 너무 크면(장 단위):** "환불 규정" 카드를 뽑았는데 배송·교환·회원가입 얘기까지 다 적혀 있습니다. 질문과 상관없는 **잡음이 함께 딸려와** LLM을 헷갈리게 하고, 토큰도 낭비돼요.
- **카드가 너무 작으면(문장 단위):** "단, 예외로 한다" 라는 카드만 뽑혔습니다. **뭐의 예외인지**가 앞 카드에 있어서 알 수 없죠. 문맥이 끊긴 겁니다.
- **적당하면(문단~절 단위):** "환불은 7일 이내 가능하며, 단 개봉 시 예외로 한다" — 하나의 완결된 규칙이 카드 한 장에. 이게 목표입니다.

## 숫자 감각

절대 정답은 없지만 실무에서 통용되는 출발점은 이렇습니다.

- **일반 문서:** 청크 300~800토큰 근처에서 시작 (짧은 FAQ는 더 작게, 논문·규정은 더 크게)
- **오버랩:** 청크 크기의 10~20%
- **평가로 조정:** 시작값으로 돌려보고, 검색 결과가 잡음이 많으면 줄이고 문맥이 끊기면 늘립니다. 청킹은 **한 번 정하고 끝이 아니라 평가와 함께 튜닝하는 값**이에요.

## 오버랩 — 경계에서 잘리는 문장 구하기

문서를 기계적으로 자르면 하필 중요한 문장이 **청크 경계에 걸쳐** 두 동강 나는 사고가 생깁니다. 그래서 이웃 청크끼리 **겹치는 구간**을 둡니다. 카드를 만들 때 앞 카드의 마지막 두어 줄을 다음 카드 첫머리에 한 번 더 적어두는 것 — 경계에 걸린 문장도 어느 한쪽 카드에는 온전히 담기게요.

```python
def chunk(text, size=600, overlap=100):
    """단순 문자 기준 청킹 (개념 시연용). 실전은 문단/문장 경계 우선."""
    out, start = [], 0
    while start < len(text):
        out.append(text[start:start + size])
        start += size - overlap        # overlap 만큼 겹치며 전진
    return out

pieces = chunk(document)
print(len(pieces), "chunks")
```

실전에서는 글자 수로 뚝뚝 자르기보다, **문단 → 문장 순으로 경계를 존중하며** 목표 크기에 맞추는 방식(재귀적 분할)이 표준입니다. 마크다운·규정문서라면 제목/조항 구조를 그대로 경계로 쓰는 게 가장 좋고요.

## 한 가지 더 — 청크에 '주소'를 달아라

같은 내용이라도 "어느 문서, 어느 장의 조각인지" 메타데이터(제목, 섹션, 날짜)를 청크에 함께 저장하면, 검색 필터링도 되고 LLM이 출처를 밝히며 답할 수 있게 됩니다. 카드 귀퉁이에 책 제목과 페이지를 적어두는 습관이에요.

## 한 줄 정리

청킹의 기준은 **"한 조각 = 하나의 완결된 생각"**. 300~800토큰에서 시작해 평가로 조정하고, 10~20% 오버랩으로 경계 사고를 막고, 메타데이터로 주소를 달아두세요. RAG 품질 문제의 절반은 모델이 아니라 **자르기**에서 옵니다.
