> **한 줄 요약:** 블로그처럼 매번 계산할 게 없는 사이트는 **완성된 HTML을 그대로 나눠주기만** 하면 됩니다. GitHub Pages가 그 배달을 공짜로 해줘요. 서버를 빌릴 필요가 없습니다.

## 이런 고민

개인 블로그 하나 띄우자고 서버 빌리고, OS 깔고, nginx 설정하고, 인증서 붙이고… 배보다 배꼽이 큽니다. 대부분의 블로그는 사실 그렇게까지 필요 없어요.

## 비유: 전단지 vs 주문 요리

- **동적 사이트**는 주문 요리입니다. 요청이 올 때마다 서버가 DB 뒤지고 페이지를 새로 조립해요. 그래서 서버가 계속 일하고, 트래픽 몰리면 느려지고, 죽기도 합니다.
- **정적 사이트**는 미리 인쇄해둔 전단지입니다. 완성본이 있으니 그냥 건네주면 끝. 계산할 게 없어서 **빠르고, 안 죽고, 공짜**에 가깝습니다.

블로그 글은 한 번 쓰면 안 바뀌죠. 그래서 정적 사이트가 딱입니다.

## GitHub Pages가 하는 일

저장소에 든 HTML 파일을 그대로 웹에 서빙해줍니다. 규칙만 알면 됩니다.

- `<사용자명>.github.io` 라는 이름의 저장소는 **푸시하면 자동으로** `https://<사용자명>.github.io/` 에 발행됩니다.
- 발행할 폴더는 Settings → Pages에서 고릅니다: 저장소 루트 `/` 또는 `/docs`.
- `index.html` 이 그 폴더 최상단에 있어야 홈으로 뜹니다.

## `.nojekyll` — 이 빈 파일 하나가 중요

GitHub Pages는 기본적으로 올라온 파일을 **Jekyll로 한 번 빌드**하려 합니다. 그런데 이미 완성된 HTML을 올리는 경우엔 그 빌드가 오히려 방해가 돼요(엉뚱한 파일에서 에러). 발행 폴더에 내용이 빈 **`.nojekyll`** 파일 하나만 두면 "빌드하지 말고 그냥 서빙해" 라는 뜻이 됩니다. 빌드 단계가 사라지니 깨질 일도 없어져요.

## 올리기 전에 로컬에서 미리보기

파이썬만 있으면 웹서버 흉내를 바로 낼 수 있습니다.

```bash
cd 사이트폴더
python3 -m http.server 8000
# 브라우저에서 http://localhost:8000 열기
```

## 흔한 함정: 404

Pages 소스로 지정한 폴더와 `index.html` 위치가 **어긋나면** 사이트가 404를 냅니다. 예를 들어 Pages는 `/docs`를 보는데 파일은 루트에 있으면 빈 걸 서빙하죠. 반대도 마찬가지입니다.

```
Settings → Pages → 지정한 폴더  ==  index.html 이 실제로 있는 폴더
```

이 둘만 일치시키면 대부분의 404는 사라집니다.

## 한 줄 정리

완성된 **HTML + `.nojekyll` + Pages 소스 폴더**, 이 셋만 맞추면 서버 0원으로 사이트가 뜹니다. 블로그엔 이게 정답에 가까워요.
