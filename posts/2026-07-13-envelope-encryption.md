> **한 줄 요약:** KMS는 데이터를 직접 암호화하지 않습니다. **데이터는 일회용 열쇠(DEK)로 잠그고, 그 일회용 열쇠를 KMS의 마스터키(KEK)로 잠가서** 데이터 옆에 같이 보관합니다. 열쇠를 봉투에 넣어 봉인한다고 해서 **봉투암호화(envelope encryption)**예요.

## 순진한 설계는 왜 안 되나

"암호화 키를 안전하게 보관하는 서비스가 KMS라며? 그럼 파일을 KMS로 보내서 암호화해달라고 하면 되겠네" — 첫 직관은 이렇습니다. 안 되는 이유가 두 가지예요.

1. **크기와 속도:** KMS API는 한 번에 암호화할 수 있는 크기가 작습니다(KB 단위). 1GB 파일을 매번 네트워크 너머 KMS로 보내는 건 느리고 비쌉니다.
2. **키가 밖으로 못 나옴이 KMS의 존재 이유:** 마스터키는 KMS(HSM) 밖으로 절대 나오지 않는 게 보안의 핵심입니다. 그렇다고 모든 데이터를 키가 있는 곳으로 가져갈 수는 없죠.

그래서 발상을 바꿉니다 — **데이터를 열쇠에게 보내지 말고, 열쇠를 데이터에게 보내자. 단, 봉인해서.**

## 비유: 금고, 상자, 그리고 봉투

- **KMS의 마스터키(KEK)** = 은행 금고 안에서 절대 안 나오는 **마스터 열쇠**
- **데이터 키(DEK)** = 상자 하나를 잠그기 위해 그 자리에서 만든 **일회용 열쇠**
- 절차: 일회용 열쇠로 상자(데이터)를 잠근다 → 일회용 열쇠를 **봉투에 넣고 마스터 열쇠로 봉인**한다 → **봉인된 봉투를 상자에 테이프로 붙여** 함께 보관한다 → 일회용 열쇠 원본은 즉시 파기

상자를 열 때는 봉투를 은행(KMS)에 가져가 "봉인 좀 풀어주세요" 하면, 은행이 신분(IAM) 확인 후 봉투만 열어줍니다. 마스터 열쇠는 여전히 금고 밖으로 안 나왔죠.

## 실제 흐름

**암호화:**

```python
# 1) KMS에 데이터 키 발급 요청 → 평문 DEK + 암호화된 DEK 두 벌을 받음
plain_dek, encrypted_dek = kms.generate_data_key(key_id="my-master-key")

# 2) 평문 DEK로 데이터를 로컬에서 암호화 (빠름 — 네트워크 불필요)
ciphertext = aes_encrypt(data, key=plain_dek)

# 3) 평문 DEK는 메모리에서 즉시 파기, 암호화된 DEK는 데이터와 함께 저장
del plain_dek
save(ciphertext, encrypted_dek)   # 봉인된 봉투를 상자에 붙여서 보관
```

**복호화:**

```python
ciphertext, encrypted_dek = load()

# 1) 봉인된 DEK만 KMS로 보내 풀어달라고 요청 (IAM 권한 검사 + 감사 로그)
plain_dek = kms.decrypt(encrypted_dek)

# 2) 로컬에서 데이터 복호화, DEK는 다시 파기
data = aes_decrypt(ciphertext, key=plain_dek)
del plain_dek
```

KMS를 오가는 건 **몇십 바이트짜리 DEK뿐**입니다. 데이터가 1GB든 1TB든 상관없어요.

## 이 구조가 공짜로 주는 것들

- **키 회전이 가벼움:** 마스터키를 교체해도 이미 저장된 데이터를 다시 암호화할 필요가 없습니다. 봉투(DEK)만 새 마스터키로 다시 봉인하면 되니까요.
- **폭발 반경 제한:** DEK 하나가 유출돼도 그 상자 하나만 위험합니다. 마스터키는 무사하니까요.
- **중앙 통제와 감사:** 모든 복호화가 KMS를 거치므로 "누가 언제 뭘 열었는지"가 한 곳에 기록되고, IAM으로 열람 권한을 즉시 회수할 수 있습니다. 마스터키를 비활성화하면 **모든 데이터가 한 번에 잠기는** 킬 스위치도 되고요.

## NCP / AWS 둘 다 같은 그림

AWS KMS의 `GenerateDataKey`/`Decrypt`, NCP KMS(Key Management Service)의 데이터 키 발급 — 이름만 다르지 봉투암호화 구조는 동일합니다. S3/Object Storage의 서버측 암호화(SSE-KMS)도 내부적으로 이 그림이 자동으로 돌아가는 것이고요. Secret Manager가 비밀을 저장할 때 밑에서 쓰는 것도 결국 KMS입니다 — 금고 위의 금고인 셈이죠.

## 한 줄 정리

봉투암호화는 **"데이터는 일회용 열쇠로, 일회용 열쇠는 마스터키로"** 잠그는 2단 구조입니다. 큰 데이터를 빠르게 암호화하면서도 마스터키는 금고 밖으로 한 발짝도 안 나오게 하는, KMS 설계의 핵심 트릭이에요.
