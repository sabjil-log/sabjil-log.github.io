> **한 줄 요약:** 서버는 잘 도는데 로드밸런서가 UNHEALTHY라고 우기면, 범인은 거의 항상 **① 체크 포트가 다르거나 ② 체크 경로가 200을 안 주거나 ③ 방화벽이 LB의 노크를 막거나** — 이 셋 중 하나입니다.

## 증상

배포는 성공했고 서버에 SSH로 들어가 `curl localhost` 하면 잘 나옵니다. 그런데 LB 콘솔엔 타겟이 계속 **UNHEALTHY**, 트래픽은 안 들어오죠. "서버 멀쩡한데 왜?"의 전형적인 상황입니다.

## 비유: 경비원의 노크

로드밸런서는 몇 초마다 각 서버 방문을 **노크(헬스체크)** 하는 경비원입니다. 정해진 **문(포트)** 을, 정해진 **암호(경로)** 로 두드리고, 정해진 **대답(상태코드 200)** 이 와야 "살아있음" 도장을 찍어요. 셋 중 하나만 어긋나도 방에 사람이 있건 말건 "없음" 처리합니다.

## 1번 용의자 — 포트가 다르다

헬스체크 포트와 앱이 실제로 듣는 포트가 다른 경우. 특히 앱은 8080인데 체크는 80으로 가는 식이죠.

```bash
# 서버에서: 앱이 어느 포트/주소로 듣는지
ss -tlnp | grep -E ':(80|8080|443)'
```

여기서 `127.0.0.1:8080` 으로만 떠 있다면 그 자체가 문제입니다 — LB는 외부에서 오니까 **`0.0.0.0` 바인딩**이어야 해요. 콘솔의 타겟그룹 포트 설정과 이 출력이 일치하는지 맞춥니다.

## 2번 용의자 — 경로가 200을 안 준다

체크 경로(`/health` 등)가 404를 주거나, 인증에 걸려 401/302(리다이렉트)를 주는 경우. LB는 보통 **200만** 성공으로 봅니다(설정에 따라 2xx~3xx 허용 가능).

```bash
# 서버 안에서 체크 경로를 직접 흉내
curl -i http://127.0.0.1:8080/health
# HTTP/1.1 200 OK 가 아니면 여기가 범인
```

HTTPS 체크인데 서버가 HTTP만 하거나(프로토콜 불일치), `/health`가 DB까지 찔러서 느려 **타임아웃**(보통 수 초)에 걸리는 것도 이 계열입니다. 헬스 엔드포인트는 가볍게, 즉시 200만 주게 만드세요.

## 3번 용의자 — 방화벽이 노크를 막는다

가장 자주 놓칩니다. 헬스체크는 **LB(의 내부 IP 대역)에서** 옵니다. 서버 방화벽(ACG/보안그룹)이 그 출발지를 안 열어놨으면 노크 자체가 도착을 못 해요.

- **ACG/보안그룹(스테이트풀):** 인바운드에 **LB가 있는 서브넷 대역**(또는 벤더가 안내하는 헬스체크 소스)을 체크 포트로 허용.
- **NACL(스테이트리스):** 여기가 진짜 함정. 인바운드만 열면 안 되고, 응답이 나가는 **아웃바운드 임시 포트(1024-65535)** 도 열어야 합니다. 안 열면 "노크는 들리는데 대답이 문밖으로 못 나가는" 상태 — 간헐적/전체 UNHEALTHY의 단골 원인.

```bash
# 서버에서: 노크가 오긴 하는지 눈으로 확인 (체크 포트 기준)
sudo tcpdump -ni any port 8080
# LB 대역에서 SYN이 주기적으로 보이면 방화벽 통과는 OK → 1·2번으로
# 아무것도 안 보이면 → 방화벽(3번)이 범인
```

## 진단 순서 요약

`ss`로 포트·바인딩 확인 → `curl 127.0.0.1`로 경로·상태코드 확인 → `tcpdump`로 노크 도착 여부 확인. 이 세 명령이면 3대장 중 누가 범인인지 5분 안에 갈립니다.

## 한 줄 정리

헬스체크 실패는 서버 문제가 아니라 **약속 불일치**입니다. **포트·경로(200)·방화벽(특히 스테이트리스 NACL의 리턴 포트)** — 노크가 닿고, 대답이 200으로 돌아가는지만 맞추면 끝나요.
