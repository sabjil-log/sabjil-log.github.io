> **한 줄 요약:** 컨테이너의 파일시스템은 **컨테이너가 지워지면 함께 사라집니다.** 살아남아야 할 데이터(DB, 업로드 파일)는 반드시 **볼륨**이나 **바인드 마운트**로 밖에 두세요. 운영 데이터는 볼륨, 개발 중 소스코드는 바인드 마운트 — 이 구분이 기본기입니다.

## 사고 현장

디스크 정리한다고 `docker rm` 몇 번, 혹은 `docker compose down` 후 다시 `up`. 그런데 DB가 텅 비어 있습니다. 백업은... 없죠. 원인은 단순합니다 — **데이터가 컨테이너 안에** 있었던 거예요.

## 비유: 호텔 방과 개인 창고

컨테이너는 **호텔 방**입니다. 체크아웃(rm)하면 방은 초기화돼요. 방 안 서랍에 넣어둔 물건(컨테이너 내부에 쓴 파일)은 함께 사라집니다.

살아남으려면 물건을 **방 밖**에 둬야 합니다. 방법이 둘이에요.

- **볼륨(volume):** 호텔이 운영하는 **개인 창고**. 도커가 관리하는 전용 공간(`/var/lib/docker/volumes/...`)에 데이터를 두고, 방에는 그 창고로 통하는 문만 냅니다. 방을 몇 번 바꿔도 창고는 그대로.
- **바인드 마운트(bind mount):** **내 집 방 하나를 호텔 방에 직결**하는 것. 호스트의 특정 경로(`/home/me/app`)를 컨테이너 안에 그대로 보여줍니다.

```bash
# 볼륨: 도커가 관리하는 이름 있는 창고
docker run -d -v pgdata:/var/lib/postgresql/data postgres

# 바인드 마운트: 호스트 경로를 직접 연결 (경로가 / 로 시작하면 바인드)
docker run -d -v /home/me/site:/usr/share/nginx/html nginx
```

구분법은 간단합니다 — **`-v` 첫 부분이 이름이면 볼륨, 경로면 바인드.**

## 언제 뭘 쓰나

**볼륨 = 운영 데이터.** DB 데이터, 업로드 파일처럼 "컨테이너보다 오래 살아야 하는 것". 도커가 권한·위치를 관리해줘서 이식성이 좋고, 백업 대상도 명확해집니다.

**바인드 마운트 = 개발 편의.** 소스코드를 물려두면 에디터로 고치는 즉시 컨테이너에 반영 — 개발 루프의 핵심이죠. 설정 파일 하나만 꽂을 때도 씁니다. 단점은 호스트 경로에 종속되고(다른 서버에선 그 경로가 없을 수 있음), 호스트 파일을 컨테이너가 직접 만지므로 권한(UID 불일치) 문제가 따라온다는 것.

compose로 쓰면 이렇게 나뉩니다.

```yaml
services:
  db:
    image: postgres
    volumes:
      - pgdata:/var/lib/postgresql/data     # 볼륨 (운영 데이터)
  app:
    build: .
    volumes:
      - ./src:/app/src                       # 바인드 (개발 소스)

volumes:
  pgdata:                                    # 이름 있는 볼륨 선언
```

## 함정 셋

**① `docker compose down -v`** — 이 `-v` 는 **볼륨까지 삭제**합니다. "정리 좀 하자"고 습관처럼 붙였다가 DB가 증발하는 단골 사고. down은 그냥 down, 볼륨 삭제는 의식적으로.

**② 이미지에 데이터가 있다는 착각** — `docker commit`이나 이미지 재빌드로 데이터가 보존될 거라 기대하면 안 됩니다. 데이터는 이미지 레이어가 아니라 (사라지는) 컨테이너 레이어에 쓰입니다.

**③ 어디에 뭘 마운트해야 하는지 모름** — 이미지마다 데이터 경로가 다릅니다(postgres는 `/var/lib/postgresql/data`, mysql은 `/var/lib/mysql`). Docker Hub 문서의 "Where to Store Data" 섹션이 항상 답을 갖고 있어요.

## 점검과 수습

```bash
docker volume ls                                   # 창고 목록
docker inspect 컨테이너 --format '{{json .Mounts}}' | python3 -m json.tool
# 지금 이 컨테이너의 데이터가 어디 붙어 있는지 — 비어 있으면 위험 신호

# 볼륨 백업 (창고 통째로 tar)
docker run --rm -v pgdata:/data -v $(pwd):/backup alpine \
  tar czf /backup/pgdata-$(date +%Y%m%d).tar.gz -C /data .
```

이미 컨테이너를 지웠다면 — 컨테이너만 지웠고(`docker rm`) 볼륨을 쓰고 있었다면 `docker volume ls`에 창고가 남아 있을 수 있습니다. 볼륨 없이 컨테이너 내부에 있었다면, 안타깝지만 복구 수단이 사실상 없습니다. 그래서 이 글이 사고 **전에** 읽혀야 하는 글이에요.

## 한 줄 정리

**컨테이너는 언제든 버려질 수 있다는 전제**로 설계하세요. 살아남을 데이터는 볼륨에, 개발 소스는 바인드로, `down -v`는 손가락 조심 — 그리고 `inspect .Mounts`로 지금 어디에 데이터가 있는지 오늘 확인해두세요.
